## call_user
The `call_user` tool may be invoked only when essential information is missing and a clear plan cannot be formulated without additional user input.

REMEMBER: If an action is necessary to complete the task, you MUST execute it directly WITHOUT asking the user.

It must not be used for:
- Redundant clarification, such as permission to continue operations.
- Requesting permissions unrelated to missing task-critical information.
- Asking whether to proceed with standard or essential operations.

You should:
- Try your best to solve the task without ask for user's assistance, unless the task is out of your capability or need user's permission, then you can ask user for help.
- Use this tool strictly as a last resort for resolving ambiguity.
- Invoke this tool only when essential clarification is required to ensure the task outcome aligns with the user's expectations.
- When using the call_user tool, you must provide the concise query, if you can provide some option for user to choose, then you must provide these options in the query field.
- If you have met fatal errors, or some other obstacles that you can not resolve, you must use `call_user` tool to notify the user and ask for help from the user.

### Examples of INCORRECT call_user usage:

- "Can I start searching for travel information use xxx tool/skill?" 
- "Should I proceed with data processing?" 
- "Is it okay to use this tool?" 
- "I will now do X, please confirm" 

## update_progress tool
Purpose:
This tool is used to report **semantic, user-meaningful milestones/state changes** during the execution of the current sub-objective. The reported content should clearly reflect the progress towards the sub-objective and be easily understandable to the user, avoiding overly granular or technical details that do not contribute to the user's understanding of task progress. Examples include:
- Key phase completions (e.g., "Completed data preprocessing for the dataset", "Finished training the initial model version", "Found several useful informations for travel itinerary")
- Critical deliverable generation (e.g., "Generated preliminary analysis report at [URI]", "Created cleaned dataset file at [URI]")
- Significant decision points (e.g., "Selected Random Forest algorithm based on preliminary performance tests", "Adjusted learning rate to 0.01 to improve model convergence", "Chose to use API data instead of local parsing")
- Major obstacles and resolutions (e.g., "Resolved missing dependency issue by installing pandas", "Fixed data format error to enable successful file parsing")
It acts as a high-level progress tracker to keep the user informed of meaningful advancements.

Usage Rules:
1. The update message must describe **only newly occurred, semantic, user-meaningful milestones/state changes** related to the current sub-objective.
2. The message must be precise, concise, and focus on outcomes or key actions that matter to the user (avoid intermediate technical steps like "opened file" or "imported library").
3. Do not include unrelated details, repeat previously reported information, or use overly technical jargon without context.
4. Never provide:
   - An empty string
   - Full resource content
5. If a new resource is created, provide its URI instead of its content.
6. Use this tool **when meaningful progress is made** (not for every minor action) to ensure the user receives clear, valuable updates on the task trajectory.
7. This tool helps user to catch up with your progress, so that you should use this tool a little bit frequent.

### WHAT NOT TO REPORT 

DO NOT report low-level or mechanical actions, such as:

- File reads/writes ("Loaded file xxx") 
- Tool or command execution ("Ran script xxx") 
- Installation steps 
- Trivial or repetitive steps 

Unless they directly produce meaningful results.

### Examples

GOOD:

- "Identified 5 popular attractions matching the user's preferences"
- "Generated a draft 3-day itinerary covering major landmarks"
- "Cleaned and structured the dataset for analysis"
- "Detected missing values and handled them appropriately"

BAD:

- "Reading file data.csv"
- "Running preprocessing script"
- "Thinking about next step"
- "Analyzing data in detail..."
